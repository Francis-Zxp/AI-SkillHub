# AI SkillHub V2 产品需求文档 PRD

文档版本：v0.1  
更新日期：2026-05-27  
产品阶段：V2 app-next 原型线，v1 维护线保留  
目标读者：产品、UI/UX 设计、前端、后端、测试、后续 AI 设计工具输入

## 1. 产品一句话

AI SkillHub 是一个中文友好的 AI Agent 能力管理中枢，用来统一管理 Skills、Prompt 资料、GitHub 来源、本地/zip 导入、AI 工具接管、工作区、备份、诊断和发布闸门。

它不是单纯的“skills 文件夹链接工具”，而是要成为普通用户、科研用户、设计师、开发者和实验室团队管理 AI Agent 能力包的桌面软件。

## 2. PRD 使用原则

这份 PRD 是设计和开发的统一依据。

- 设计工具可以根据本文件生成界面、交互稿、组件库和视觉方向。
- 开发实现必须按本文件拆分功能、状态、异常和验收标准。
- 测试验收必须覆盖正常流程、空状态、错误状态、未安装工具、权限不足、路径含中文/空格、没有 Git、没有 Codex 等真实场景。
- 后续如果需求变化，必须更新本文件或补充变更记录，不能只在聊天里口头修改。

## 3. 背景与问题

当前 AI 用户会从 GitHub、博客、论坛、团队成员那里收集大量 Skills 和 Prompt。不同工具有不同目录，例如 Claude Code、Codex、Antigravity、Cursor、Windsurf 等。普通用户经常遇到这些问题：

1. 不知道下载的仓库里哪个目录才是真正的 Skill。
2. 不知道 `SKILL.md`、Prompt、README、MCP、Agent 配置的区别。
3. 同一个 Skill 要分别复制到多个 AI 工具目录，容易乱。
4. GitHub 来源更新后，本地技能不会自动保持最新。
5. Windows 上软链接、权限、中文路径、空格路径、Git、WebView2 都可能出错。
6. 分享给别人时，对方可能没有 Codex、没有 Git、没有 WebView2，软件应该给出清楚提示，而不是抛出技术栈错误。
7. 用户害怕误删自己的技能目录，需要备份、恢复、回滚和诊断包。

AI SkillHub V2 的目标是把这些混乱统一成一个安全、清晰、可分享、可扩展的桌面产品。

## 4. 产品目标

### 4.1 核心目标

1. 让用户用一个软件管理所有 AI Agent 能力来源。
2. 自动识别 Skill、Prompt、普通资料、风险内容和重复项。
3. 支持 Claude Code、Codex、Antigravity 等 AI 工具的全局和项目级能力管理。
4. 支持备份、恢复、回滚、诊断、分享验证和发布闸门。
5. 让非技术用户能看懂“现在能不能同步 / 能不能发布 / 为什么不能”。
6. 保留高级用户未来需要的 CLI、批量部署、Git 备份和团队协作扩展空间。

### 4.2 体验目标

1. 双击即可打开正式版软件。
2. 首次打开能自动体检环境，不创建假目录。
3. 出错时显示人话说明和下一步建议，不显示原始技术栈。
4. 界面清晰、克制、专业，有现代桌面软件质感。
5. 中文优先，同时支持 English 和 한국어。
6. 高 DPI 下文字清晰，不模糊、不发虚、不依赖过度毛玻璃。

### 4.3 成功指标

| 指标 | 目标 |
|---|---|
| 首次启动成功率 | 95% 以上 |
| 没装 Codex / Git / WebView2 时 | 软件正常打开，并给出清楚提示 |
| GitHub 来源添加成功率 | 合法普通 GitHub 仓库 URL 95% 以上可处理 |
| 诊断包导出 | 30 秒内完成，不包含隐私 token |
| 真实同步前 | 必须先显示备份与回滚状态 |
| 普通用户理解成本 | 首页 10 秒内能看懂当前状态 |

## 5. 用户角色

### 5.1 新手用户

特点：

- 会下载 GitHub zip，但不懂目录结构。
- 不知道 Skill、Prompt、MCP 的区别。
- 害怕命令行。

核心需求：

- 粘贴 GitHub 地址即可添加。
- 选择本地 zip 后先预览，再确认导入。
- 软件告诉他“这个不是 Skill，只是 Prompt 资料”。
- 出错时给出中文解释。

### 5.2 科研 / 论文用户

特点：

- 收集论文写作、文献综述、科研绘图、数据分析类 Skills。
- 可能在实验室多台电脑或多人之间共享。

核心需求：

- 按论文、图表、数据、投稿、审稿等标签整理。
- 支持 Preset，一键启用“论文写作套装”。
- 支持分享给师兄弟，并能稳定部署。

### 5.3 设计 / Vibe Coding 用户

特点：

- 关注 UI、品牌、Prompt、设计工作流。
- 需要管理设计类 Skills 和 Prompt。

核心需求：

- Prompt 和 Skill 分开管理。
- 可按 UI 设计、品牌、图片生成、前端实现等标签查找。
- 界面本身要足够精致，不能像临时脚本。

### 5.4 开发者 / 高级用户

特点：

- 会用 Git、CLI、Tauri、Node、Rust。
- 需要批量管理、自动化和可恢复操作。

核心需求：

- SQLite 数据透明。
- CLI doctor / sync / audit / export。
- Git 备份、快照、回滚、日志。
- 项目级工作区和 AGENTS.md 生成。

## 6. 产品定位边界

### 6.1 V2 必须做

1. Tauri 桌面软件。
2. React + TypeScript UI。
3. Rust 后端。
4. SQLite 本地数据库。
5. 双击 exe 启动正式版。
6. 中央 Skill Library。
7. GitHub / 本地 / zip 来源管理。
8. Skill / Prompt / 普通资料区分。
9. Claude Code、Codex、Antigravity 基础适配。
10. 工作区模型：全局、Agent、项目。
11. Preset 和标签。
12. 诊断包。
13. 备份 dry-run、恢复 dry-run、回滚锁。
14. Release Readiness 发布闸门。
15. 多语言和主题。

### 6.2 V2 早期不做

1. 不做云账号系统。
2. 不做在线商业 Marketplace。
3. 不默认上传用户 Skills。
4. 不自动覆盖 AI 工具目录。
5. 不绕过系统权限创建危险软链接。
6. 不把 Prompt 仓库强行当成 Skill。
7. 不在没有备份和回滚前执行真实恢复。

### 6.3 未来可做

1. skills.sh / GitHub 推荐源市场。
2. MCP 管理。
3. Agent / Command / Rule 管理。
4. CLI 批量部署。
5. 团队共享仓库。
6. Skill 安全评分。
7. Skill 使用建议和质量评估。

## 7. 启动方式需求

### 7.1 当前开发阶段

开发阶段允许使用命令启动：

- `pnpm dev:desktop`
- `pnpm tauri dev`

原因：开发时需要 Vite 热更新、Rust 编译和调试日志。

### 7.2 正式用户阶段

正式版本必须支持双击启动。

最低要求：

1. 用户下载 release 包。
2. 解压或安装。
3. 双击 `AI SkillHub.exe`。
4. 软件打开并自动体检环境。
5. 没装任何 AI 编程工具时，显示“未检测到可接管的 AI 工具”，但软件仍可用于收集来源和管理中央库。

### 7.3 发布包形态

优先级从高到低：

1. Windows 安装包：`.msi` 或 `.exe installer`。
2. 免安装 zip：包含 `AI SkillHub.exe` 和必要资源。
3. 开发调试 exe：仅给开发者，不作为分享版本。

### 7.4 启动验收标准

| 场景 | 预期 |
|---|---|
| 用户双击正式 exe | 3 秒内出现主窗口或加载页 |
| 没有 Codex | 显示未检测到 Codex，不报错 |
| 没有 Claude Code | 显示未检测到 Claude Code，不报错 |
| 没有 Git | 来源更新功能提示需要安装 Git，其他页面可用 |
| 路径含中文和空格 | 正常读取配置和数据库 |
| WebView2 缺失 | 给出下载/修复指引 |
| SQLite 文件不存在 | 自动初始化 |

## 9. 核心对象模型

### 9.1 Source 来源

来源是用户添加的 GitHub 仓库、本地文件夹、zip 包或 Prompt 资料。

字段：

- 名称
- 类型：GitHub / local / zip / prompt / unknown
- URL 或本地路径
- 分类
- 标签
- 备注
- 当前状态：未下载、可用、需要更新、失败、只读、Prompt 资料
- 最近同步时间

### 9.2 Skill 技能

只有包含 `SKILL.md` 或 `skill.md` 的目录才是 Skill。

字段：

- 名称
- 描述
- 来源
- 分类
- 标签
- 路径
- 健康状态
- 风险提示
- 重复状态
- 启用状态

### 9.3 Prompt 资料

Prompt 不是 Skill，不应安装到 AI 工具的 skills 目录。

字段：

- 名称
- 内容类型
- 来源
- 标签
- 备注
- 可复制 / 可收藏 / 可关联到 Preset

### 9.4 Agent Adapter

代表一个 AI 工具，例如 Claude Code、Codex、Antigravity、Cursor。

字段：

- 名称
- 是否已安装
- 默认技能目录
- 是否已接管
- 是否可写
- 同步模式：symlink / copy / disabled
- 风险状态

### 9.5 Workspace 工作区

工作区是能力生效范围。

类型：

- 全局工作区：中央库。
- Agent 工作区：某个 AI 工具的全局 Skills。
- 项目工作区：某个项目目录下的 Skills / AGENTS.md / CLAUDE.md。

### 9.6 Preset 预设

Preset 是一组 Skills / Prompt 的组合。

例子：

- 论文写作
- 科研绘图
- UI 设计
- 安全审计
- 代码质量
- 实验室新电脑部署

## 10. 关键用户流程

### 10.1 首次启动

流程：

1. 用户双击打开软件。
2. 软件自动初始化 SQLite。
3. 自动检测 Git、WebView2、Claude Code、Codex、Antigravity。
4. 显示总览状态。
5. 如果没有 AI 工具，显示“你仍然可以先添加来源，稍后再接管 AI 工具”。

异常：

- SQLite 初始化失败：提示数据库位置和修复建议。
- 权限不足：提示以普通模式可读，涉及接管时再请求权限。
- 没有 Git：GitHub 更新不可用，但本地导入可用。

### 10.2 添加 GitHub 来源

流程：

1. 用户粘贴 GitHub URL。
2. 系统校验 URL。
3. 克隆或更新到 sources 目录。
4. 扫描 `SKILL.md`。
5. 识别 Skill 数量、Prompt 内容、普通资料。
6. 显示预览。
7. 用户确认加入中央库。

规则：

- 只接受普通 GitHub 仓库 URL 或明确支持的 shorthand。
- 不接受带脚本注入、奇怪协议、凭据泄露的 URL。
- 如果仓库没有 Skill，应保留为 Prompt/资料，不安装为 Skill。

异常：

- 网络失败：提示检查网络或代理。
- 仓库不存在：提示 URL 可能错误。
- 私有仓库：提示需要凭据或手动克隆。
- 路径已存在：提示更新或重命名。

### 10.3 本地 / zip 导入

流程：

1. 用户选择文件夹或 zip。
2. 系统只读扫描。
3. 展示发现的 Skills、Prompt、普通文件。
4. 检查重复名和风险文件。
5. 用户确认导入。
6. 导入中央库并记录历史。

安全规则：

- zip 必须防 Zip Slip。
- 不执行 zip 内任何脚本。
- 不跟随不安全 symlink。
- 不覆盖已有目录，除非用户明确选择并有备份。

### 10.4 管理技能库

用户可以：

- 搜索 Skill。
- 按分类、标签、来源、健康状态过滤。
- 查看详情。
- 查看 `SKILL.md` 摘要。
- 查看风险提示。
- 查看重复项。
- 给 Skill 添加标签和备注。

不能直接做：

- 不能从详情页直接危险删除真实目录。
- 不能绕过备份直接覆盖 AI 工具目录。

### 10.5 接管 AI 工具

流程：

1. 系统显示已检测到的 AI 工具。
2. 用户选择是否接管。
3. 系统展示将要接管的目录。
4. 执行备份 dry-run。
5. 用户确认后才能真实接管。
6. 接管后显示同步模式和状态。

规则：

- 未安装工具显示“未检测到”，不创建假目录。
- 检测到但未接管显示“可接管”。
- 接管前必须有备份路径和恢复方案。

### 10.6 工作区管理

全局工作区：

- 管理中央库。

Agent 工作区：

- 展示每个 AI 工具启用的 Skills。
- 支持按 Preset 应用。

项目工作区：

- 扫描项目技术栈。
- 检测 AGENTS.md、CLAUDE.md、README.md。
- 生成只读说明草稿。
- 未来可生成项目级 AI 指令文件，但必须先预览。

### 10.7 快照与恢复

流程：

1. 用户进入快照页。
2. 系统展示最新快照和备份目标。
3. 用户执行备份 dry-run。
4. 系统展示会备份哪些目录、跳过哪些目录、阻断哪些目录。
5. 恢复 dry-run 展示会恢复哪些目标。
6. 所有条件满足后，真实恢复按钮才可解锁。

规则：

- 默认锁定真实恢复。
- 没有备份不能恢复。
- 未接管工具不恢复。
- 路径越界必须阻断。

### 10.8 发布闸门

Release Gate 必须汇总：

- 诊断体检
- 发布预检
- 分享验证
- zip 预览
- 桌面 QA
- 备份 dry-run
- 恢复 dry-run
- 回滚锁

状态：

- 全部通过：候选可复查。
- 有待复查：暂不发布。
- 有阻断项：禁止发布。

规则：

- Release Gate 只给结论，不直接打包。
- 打包入口必须在所有闸门通过后才允许出现。

## 11. 页面需求

### 11.1 总览 Dashboard

目标：让用户 10 秒内看懂软件状态。

必须显示：

- 已启用 Skills 数量。
- 来源数量。
- Prompt 数量。
- 已检测 AI 工具数量。
- 需要关注的问题数量。
- 最近同步时间。
- Release Readiness 简短状态。

主要操作：

- 添加来源。
- 立即体检。
- 打开诊断包。
- 进入发布闸门。

空状态：

- 没有来源时，引导添加 GitHub / 本地 / zip。
- 没有 AI 工具时，提示仍可先整理中央库。

### 11.2 技能库 Skill Library

必须包含：

- 搜索框。
- 标签过滤。
- 分类过滤。
- 来源过滤。
- 健康状态过滤。
- 表格 / 卡片视图切换。
- Skill 详情抽屉。

Skill 卡片展示：

- 名称
- 描述
- 标签
- 来源
- 健康状态
- 是否重复
- 是否启用

### 11.3 来源 Sources

必须包含：

- GitHub URL 输入。
- 类型选择：Skill / Prompt / 自动识别。
- 分类和标签。
- 备注。
- 添加并扫描。
- 本地文件夹导入。
- zip 导入。
- 来源列表。
- 来源详情。

来源状态：

- 正常
- 需要更新
- 未下载
- 无 Skill，作为 Prompt/资料保留
- URL 错误
- 网络失败
- 权限失败

### 11.4 工作区 Workspaces

必须包含：

- 全局工作区。
- Agent 工作区。
- 项目工作区。
- 工作区详情。
- 项目扫描结果。
- 只读说明预览。

交互：

- 点击工作区卡片进入详情。
- 启用/停用只写入 SQLite 状态，早期不触碰真实目录。

### 11.5 AI 工具 Agents

必须包含：

- Claude Code
- Codex
- Antigravity
- Cursor
- Windsurf
- GitHub Copilot
- OpenCode
- Gemini CLI
- 自定义工具路径

每个工具展示：

- 是否检测到
- 技能目录
- 是否接管
- 是否可写
- 同步模式
- 风险状态
- 下一步建议

### 11.6 快照 Snapshots

必须包含：

- 最新快照。
- 写入闸门。
- 备份目标清单。
- 备份 dry-run。
- 恢复 dry-run。
- 回滚计划。
- 历史快照。

按钮状态：

- 没有备份时，恢复按钮禁用。
- 有阻断目标时，真实同步按钮禁用。
- 所有危险按钮必须有确认弹窗。

### 11.7 发布闸门 Release Gate

必须包含：

- Release Readiness 总结卡。
- 闸门总数。
- 已通过数。
- 待复查数。
- 阻断数。
- 每个闸门详情。
- 最近报告时间。
- 下一步建议。

文案要求：

- 不写“失败了”这种笼统话。
- 必须写“为什么不能发布”和“下一步做什么”。

### 11.8 设置 Settings

必须包含：

- 中央库路径。
- 数据库路径。
- 主题。
- 字号。
- 语言。
- 同步模式默认值。
- Release / Build Guide。
- Desktop QA Checklist。
- 诊断包导出。
- 开发者信息。

## 12. 交互规则

### 12.1 按钮状态

| 状态 | 规则 |
|---|---|
| 默认 | 可点击，明确动词 |
| 加载中 | 显示 spinner 或进度，不允许重复点击 |
| 禁用 | 必须显示禁用原因 |
| 危险操作 | 红色/警示色，必须二次确认 |
| 成功 | toast + 状态更新 |
| 失败 | 人话错误 + 下一步建议 + 诊断入口 |

### 12.2 弹窗规则

弹窗只用于：

- 危险确认。
- 导入预览确认。
- 权限/路径选择。
- 重要错误处理。

弹窗不应滥用。普通状态变化用 toast 或页面内状态。

### 12.3 错误文案规则

错误文案结构：

1. 发生了什么。
2. 为什么可能发生。
3. 你下一步可以做什么。
4. 技术详情放进诊断包，不直接吓用户。

示例：

- 不好：`RuntimeException at SkillHub.ps1 line 203`
- 好：`这个 GitHub 地址格式不符合要求。请粘贴普通仓库地址，例如 https://github.com/owner/repo.git。详细日志已写入诊断包。`

## 13. UI/UX 设计要求

### 13.1 视觉方向

关键词：

- 清晰
- 现代
- 轻量
- 可信
- 中文友好
- 专业但不冰冷
- 有细节但不花哨

不允许：

- 过度毛玻璃导致文字发虚。
- 大面积低对比浅色文字。
- 全屏巨型字号。
- AI 生成感很重的随机彩色标签。
- 模仿竞品品牌。
- 把界面做成营销落地页。

### 13.2 布局建议

桌面默认窗口：

- 建议默认尺寸：1280 x 820 到 1440 x 900。
- 左侧固定导航：220 到 260 px。
- 主内容最大宽度：不要无限拉伸，建议 1180 到 1360 px。
- 页面标题区简洁，保留主要动作。
- 卡片用于具体对象，不要卡片套卡片。
- 表格适合高密度信息，卡片适合概览和工作区。

### 13.3 字体

Windows 中文优先：

- `Microsoft YaHei UI`
- `Segoe UI`
- `Inter`
- 系统默认 sans-serif

要求：

- 正文 14 到 15 px。
- 小字不低于 12 px。
- 页面标题不要过大。
- 字重层级清晰。
- 不用负 letter-spacing。

### 13.4 色彩

默认主题建议：清新浅色专业风。

主色：

- Mint / Teal：用于成功、已连接、主操作。
- Sky / Cyan：用于信息、链接、可探索内容。
- Soft Amber：用于待复查。
- Coral / Rose：用于阻断和危险操作。

规则：

- 标签颜色必须有语义，不要随机。
- 重要状态颜色保持一致。
- 深色主题必须保证对比度。

### 13.5 动效

可用：

- hover 轻微位移或阴影。
- 切换页面淡入。
- 状态胶囊轻微过渡。
- 导入扫描进度动画。
- 成功操作 toast。

避免：

- 大面积炫光。
- 持续闪烁。
- 影响阅读的背景动画。
- 每个组件都动。

### 13.6 Logo 和品牌

必须保留 AI SkillHub 原创 Logo 和名称。

设计可以学习优秀项目的信息架构，但不能复制竞品视觉、图标、品牌和页面布局。

## 14. 主题需求

至少支持：

1. 默认清新主题
2. 简洁浅色主题
3. 深色主题
4. 跟随系统

主题切换要求：

- 立即生效。
- 写入本地设置。
- 不影响数据。
- 表格、卡片、弹窗、toast、导航都必须适配。

## 15. 多语言需求

支持：

- 中文
- English
- 한국어

规则：

- 中文状态下非必要不显示英文。
- 专业名词可保留，如 Skill、Prompt、Release Gate，但需要中文解释。
- 弹窗、错误、按钮、空状态都必须跟随语言。

## 16. 安全与隐私

必须做到：

1. 诊断包脱敏。
2. 不上传用户 Skills。
3. 不记录 token 明文。
4. 不执行第三方仓库脚本。
5. zip 导入防路径穿越。
6. GitHub URL 校验。
7. 真实写入前必须备份 dry-run。
8. 真实恢复前必须恢复 dry-run。
9. 没有备份不允许恢复。
10. 未安装工具不创建假目录。

## 17. 性能要求

| 300 个 Skills 列表渲染 | 不卡顿 |
| GitHub 来源扫描 | 后台执行，UI 不阻塞 |
| 诊断包导出 | 30 秒内完成 |
| 搜索过滤 | 输入后 200 ms 内响应 |

- 高 DPI
